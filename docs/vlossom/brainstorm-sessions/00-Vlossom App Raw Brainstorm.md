## 5. Coding Conventions & Quality Bar

**Languages & Tooling (initial preference)**

- Smart contracts: `Solidity` on EVM.
- Testing: `Foundry` or `Hardhat + TypeScript` tests (ok to propose one and standardize).
- Backend/API: `TypeScript` (Node / possibly Next.js app router for API).
- DB: Postgres (or any SQL) for rich analytics & monitoring.

**Style & testing requirements**

- Write **comprehensive unit tests** for each contract.
- Include tests for:
  - reentrancy protection
  - boundary conditions (0 amount, max booking length, edge timestamps)
  - happy paths & common failure modes
- Prefer:
  - CEI pattern (checks-effects-interactions)
  - explicit events for any state changes
  - clear `require` errors (custom errors preferred in Solidity â‰¥0.8.4)

**Security mindset**

- Avoid:
  - inline assembly unless absolutely necessary
  - complex upgrade proxy patterns for v1 (only if we explicitly decide)
- If Claude proposes advanced patterns:
  - require explanation & threats model
  - keep code reviewable by human auditors
